# Debrid Services Implementation Notes

## Current provider coverage

The backend now has built-in provider clients in [`filmu_py/services/debrid.py`](../filmu_py/services/debrid.py) for:

- Real-Debrid
- AllDebrid
- Debrid-Link

Each provider currently exposes two layers of behavior:

1. **Playback refresh / unrestriction**
   - `unrestrict_link()` for provider-backed playback refresh
   - Real-Debrid also supports `refresh_download()` via provider download id

2. **Download pipeline client methods**
   - `add_magnet()`
   - `get_torrent_info()`
   - `select_files()`
   - `get_download_links()`

These methods are provider-specific and intentionally not hidden behind a fake unified API surface. The orchestration layer chooses the provider and calls the matching client explicitly.

## Runtime settings source

The provider runtime configuration now comes from [`DownloadersSettings`](../filmu_py/config.py) in [`filmu_py/config.py`](../filmu_py/config.py), which mirrors the original frontend `settings.json` `downloaders` block:

- `video_extensions`
- `movie_filesize_mb_min`
- `movie_filesize_mb_max`
- `episode_filesize_mb_min`
- `episode_filesize_mb_max`
- `proxy_url`
- `real_debrid`
- `debrid_link`
- `all_debrid`

The compatibility settings route in [`filmu_py/api/routes/settings.py`](../filmu_py/api/routes/settings.py) now reads from that structured runtime model.

## File filtering

The helper [`filter_torrent_files()`](../filmu_py/services/debrid.py) applies the downloader settings before file selection:

- extension allow-list from `video_extensions`
- movie minimum/maximum filesize
- episode minimum/maximum filesize
- `-1` maximum means no upper bound

## Rate limiting

Provider download pipeline calls now use a separate limiter bucket:

- `ratelimit:{provider}:download`

This is distinct from the playback refresh bucket:

- `ratelimit:{provider}:stream_link_refresh`

The limiter is injected into provider clients at construction time. It is not enforced as a hidden global dependency inside the client implementation.

## Worker orchestration

The worker stage in [`debrid_item()`](../filmu_py/workers/tasks.py) now performs a real provider-backed flow:

1. Load the selected stream candidate.
2. Resolve the enabled downloader provider from runtime settings.
3. Construct the provider client with the shared distributed limiter.
4. Call `add_magnet()`.
5. Poll `get_torrent_info()` until a usable provider state is reached or timeout occurs.
6. Filter candidate files with [`filter_torrent_files()`](../filmu_py/services/debrid.py).
7. Call `select_files()`.
8. Call `get_download_links()`.
9. Persist the resulting direct provider-backed download URLs into media entries through [`persist_debrid_download_entries()`](../filmu_py/services/media.py).

The worker logs propagate both `item_id` and `item_request_id` where that context is available.

## Provider selection policy

The worker now uses an explicit compatibility-first downloader priority when more than one provider is enabled:

1. Real-Debrid
2. AllDebrid
3. Debrid-Link

This is intentionally static for now:

- no load balancing
- no automatic failover
- no dynamic provider switching mid-job

If multiple providers are enabled, the worker logs a warning and still picks the highest-priority provider deterministically.

Scope note:

- Still dont have dynamic provider switching, load balancing, or broader retry-library fan-out beyond the requested `indexed` and `scraped` stage re-entry behavior.

The surrounding worker graph is now also explicitly split into:

- [`parse_scrape_results()`](../filmu_py/workers/tasks.py)
- [`rank_streams()`](../filmu_py/workers/tasks.py)
- [`debrid_item()`](../filmu_py/workers/tasks.py)
- [`finalize_item()`](../filmu_py/workers/tasks.py)

This keeps original-riven compatibility intact while making the backend stage graph cleaner for a future frontend built specifically for this backend.

## Current limitations

- The worker currently selects the first enabled provider from runtime settings; there is not yet a richer provider-ranking or fallback policy.
- Provider API response mapping is covered by focused unit tests, but there is still no live integration test suite against real upstream provider sandboxes.
- ARQ wiring now exists for the debrid stage, but the broader downloader fan-out / retry-library / finalize evolution is still incomplete.
