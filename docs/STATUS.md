# Implementation Status

## Current focus

The current priority order is:

1. Keep the merged GitHub-hosted playback-gate CI path green, validate the live `main` policy from an admin-authenticated host with the stricter enterprise profile, and keep repeated soak/provider evidence moving from one-off proof into repeatable operational evidence. The runner-readiness and policy-check paths now exist in [PLAYBACK_GATE_RUNNER_SETUP.md](PLAYBACK_GATE_RUNNER_SETUP.md), [../scripts/check_playback_gate_runner.ps1](../scripts/check_playback_gate_runner.ps1), and [../scripts/check_github_main_policy.ps1](../scripts/check_github_main_policy.ps1); Windows dev hosts are expected to fail the Linux `/dev/fuse` readiness check. The playback stability wrapper now also has an enterprise profile through [`proof:playback:gate:enterprise`](../package.json), and [`../scripts/run_playback_proof_stability.ps1`](../scripts/run_playback_proof_stability.ps1) now emits `environment_class` plus explicit `enterprise_policy` thresholds so repeated proof artifacts can be promoted into merge governance instead of remaining ad hoc.
2. Turn the persisted identity/tenant baseline into full authorization depth. Effective permissions, actor-aware API-key rotation, tenant-scoped reads, delegated tenant authorization, OIDC/JWKS bearer-token validation, persisted access-policy revisions, operator-facing `/api/v1/auth/policy`, and structured authorization decision audit logging are now present; the next step is operator-managed policy CRUD/version workflows, broader ABAC evaluation across more resources, SSO rollout docs, and stronger deny/allow retention/search.
3. Deepen the mounted FilmuVFS data plane now that chunk-engine reads, adaptive prefetch, optional L2 cache, per-handle fairness caps, global-backpressure counters, thresholded soak profiles, derived `/api/v1/stream/status` rollout-governance ratios/reasons/actions, and the enterprise soak wrapper exist: longer-running soak/backpressure validation, richer mounted metrics, and rollout hardening across more than one environment class.
4. Expand the plugin platform from the now-real runtime baseline into stronger trust/isolation and external-author distribution policy now that provenance metadata, digest validation, quarantine enforcement, trust-store-backed signature verification, publisher lifecycle/distribution/scope policy evaluation, tenancy-mode governance, and the `/api/v1/plugins/governance` trust/isolation rollup are live.
5. Extend observability and orchestration beyond the new layer-1 baseline with shipper/search-ready logs, queue-history/alerting rollups, tenant-aware identity inventory visibility, quota policy, and deeper mount/VFS data-plane visibility. Queue-history summaries, derived VFS readiness signals, tenant quota visibility plus request-intake/worker-enqueue enforcement, Redis Streams replay plus consumer-group primitives, Vector shipper config, and the log-pipeline contract check narrow the gap, but they do not yet replace a fully provisioned external search/export workflow.

The seven active enterprise operations slices now also have a machine-readable posture surface at [`GET /api/v1/operations/governance`](../filmu_py/api/routes/default.py), covering playback-gate promotion, identity/authz, tenant boundary, VFS data-plane posture, distributed control-plane readiness, SRE program status, and durable log-pipeline plus plugin-runtime-isolation readiness. This now reports OIDC validation posture, access-policy version/source, tenant-quota policy, Redis Streams replay plus consumer-group configuration, VFS chunk-cache/catalog-watch evidence, backup/restore proof automation, and log-shipper/search contract readiness, but it is still not a claim that HA failover or externally hosted search are complete.

## Current Filmu vs `riven-ts` position

- The current comparison baseline is upstream `rivenmedia/riven-ts` `main` at `e64604a`; Filmu parity and differentiation claims should be judged against that exact tree, not against older notes or the upstream `feat/rust-vfs` branch.
- Filmu is now ahead of current `riven-ts` in several important areas: the cross-process Rust sidecar VFS architecture, Windows-native WinFSP/ProjFS adapter boundary, operator-facing playback governance on `/api/v1/stream/status`, persisted OIDC/authz policy posture on `/api/v1/auth/policy`, tenant quota visibility plus first worker-pressure enforcement, plugin trust/publisher policy, and the new enterprise operations/governance surfaces.
- Filmu is no longer missing the basic mounted playback path. The Rust sidecar now has real `open`/`read`/`readdir`/`getattr`/`release` behavior, chunk-engine-backed mounted reads, adaptive prefetch, optional L2 cache, Linux `fuse3`, Windows native adapters, and validated Jellyfin/Emby/Plex proof paths; the active gap is operational hardening and merge-policy promotion, not first playback success.
- Filmu still trails the current `riven-ts` baseline in a narrower set of areas: broader plugin/package ecosystem breadth, sandboxed heavy-stage orchestration, externally provisioned searchable log infrastructure, and a more complete HA/distributed-control-plane operating model.
- The dedicated comparison sources of truth remain [`RIVEN_TS_AUDIT.md`](RIVEN_TS_AUDIT.md), [`filmcore_vs_riven_ts_audit.md`](filmcore_vs_riven_ts_audit.md), and [`filmcore_vs_ts_vfs.md`](filmcore_vs_ts_vfs.md); the top-level roadmap/TODO docs should summarize those findings rather than compete with them.
