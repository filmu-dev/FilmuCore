# FilmuCore vs Riven-TS: Full Capability Audit (March-April 2026)

## The Core Question

> Is `FilmuCore` able to run the full media acquisition pipeline end-to-end: requesting content → scraping torrents → choosing a debrid provider → managing a VFS mount → serving Plex?

**Yes on the currently validated local proof topologies, with enterprise hardening still remaining.** FilmuCore now has a real, provider-backed media acquisition pipeline with scrape → parse → rank → debrid → finalize stages backed by a built-in Torrentio scraper plus built-in Real-Debrid / AllDebrid / Debrid-Link clients, RTN-compatible ranking, durable stream/media-entry persistence, transactional outbox, scheduled recovery, tenant/authz posture, and a real Rust VFS sidecar that has been validated through Linux/WSL plus native Windows playback/provider gates. The remaining gaps are no longer first pipeline completeness or first mounted playback success; they are stronger index-stage depth, broader plugin/package breadth, sandboxed heavy-stage isolation, and enterprise operations hardening.

## Gap-Closure Roadmap: What FilmuCore Needs to Beat Riven-TS

### Phase 1 — Close the pipeline with real integrations (P1) — ✅ Largely Complete

1. ~~**Real-Debrid / AllDebrid / Debrid-Link provider clients**~~ ✅ Done — built-in clients for both playback refresh and pipeline download execution.
2. ~~**Wire ARQ tasks to real provider calls**~~ ✅ Done — `debrid_item` resolves the enabled provider, calls download-pipeline methods, persists media entries, transitions to `finalize_item`.
3. ~~**RTN-compatible ranking + parse stages**~~ ✅ Done — `parse_scrape_results` + `rank_streams` + `select_stream_candidate` exist as real ARQ worker stages with a standalone RTN compatibility package.
4. ~~**Torrentio scraper plugin**~~ ✅ Done — built-in [`TorrentioScraper`](../filmu_py/plugins/builtin/torrentio.py) now participates in the real scrape worker path.
5. **TMDB metadata worker stage** — enrichment exists for item detail/search but no dedicated `index-item` worker stage.

### Phase 2 — Productionize the VFS data plane (P1)

The core mounted path now exists. The next work is to harden and operationalize it:

1. keep mounted `open` / `read` / `readdir` / `getattr` / `release` behavior green under repeated soak and provider-pressure scenarios
2. keep the Python HTTP chunk semantics and the Rust mounted chunk semantics aligned where that improves operator reasoning and correctness
3. deepen cache correctness, prefetch fairness, abort/cancellation handling, and mounted metrics
4. promote Linux/Windows/provider proof artifacts into enforced merge policy instead of leaving them as one-off operator runs
5. decide how much broader queue-backed resolver orchestration should exist above the now-landed mount-side stale-refresh dedup baseline

### Phase 3 — Integrate the shared chunk engine + caching (P2)

Extend the byte-serving runtime around the now-implemented shared engine in [`chunk_engine.py`](../filmu_py/core/chunk_engine.py):

- wire the shared engine into direct-file/HTTP byte reads
- reuse or port the same geometry/classification semantics in mounted Rust reads
- add on-disk cache and prefetch policy once the live integration exists

### Phase 4 — Content-source and library automation depth (P2)

- keep Overseerr/Jellyseerr intake aligned with tenant/authz/quota policy
- decide whether Plex watchlist or library-update automation is worth first-class productization
- add library-update hooks only where they measurably improve operator workflow rather than as parity theater

### Phase 5 — Exceed Riven-TS quality bar

- Full scraper breadth (Jackett, Prowlarr, Mediafusion — the areas where Python Riven already beats the TS backend)
- Match the now-broader TS plugin/package platform reflected in the refreshed local working tree: `plugin-comet`, `plugin-listrr`, `plugin-mdblist`, `plugin-notifications`, `plugin-plex`, `plugin-realdebrid`, `plugin-seerr`, `plugin-stremthru`, `plugin-tmdb`, `plugin-torrentio`, `plugin-tvdb`, shared core utility packages, and the standalone `util-rank-torrent-name` package all raise the parity bar
- Operator-managed access-policy workflows, broader tenant-aware quotas/attribution, and durable operator search/export workflows
- Stream/VFS metrics (active streams, cache-hit ratio, mount latency, fallback counters)
- Cancellation /abort handling for in-progress reads
- Canary/rollback controls for VFS rollout
