# FilmuCore vs Riven-TS: Full Capability Audit (March-April 2026)

## The Core Question

> Is `FilmuCore` able to run the full media acquisition pipeline end-to-end: requesting content → scraping torrents → choosing a debrid provider → managing a VFS mount → serving Plex?

**Partially. FilmuCore now has a real, provider-backed media acquisition pipeline** with scrape → parse → rank → debrid → finalize stages backed by a built-in Torrentio scraper plus built-in Real-Debrid / AllDebrid / Debrid-Link clients, RTN-compatible ranking, durable stream/media-entry persistence, transactional outbox, and scheduled recovery. The remaining gaps are: full Plex/Emby playback validation on top of the new Rust VFS mount path, mount-side chunk-engine adoption, and content-source intake (Overseerr/Plex webhooks).

5. **TMDB metadata worker stage** — enrichment exists for item detail/search but no dedicated `index-item` worker stage.

### Phase 2 — Build the VFS mount worker (P1)

Wire FUSE on top of the existing [byte_streaming.py](file:///e:/Dev/Filmu/FilmuCore/filmu_py/core/byte_streaming.py) substrate:

1. Real [open](file:///E:/Dev/Triven_riven-fork/Triven_backend%20-%20ts/apps/riven/lib/vfs/operations/open.ts#54-174) — call link resolver, store URL in handle, and reuse the shared chunk engine's file-chunk calculations.
2. Real [read](file:///E:/Dev/Triven_riven-fork/Triven_backend%20-%20ts/apps/riven/lib/vfs/operations/read.ts#28-171) — reuse the shared engine's header/footer/body-read/cache-hit detection plus fetch/stitch behavior.
3. Real [readdir](file:///e:/Dev/Filmu/FilmuCore/filmu_py/core/byte_streaming.py#381-385) — walk registered path tree using the existing [readdir_for_path_id](file:///e:/Dev/Filmu/FilmuCore/filmu_py/core/byte_streaming.py#381-385) wrapper.
4. Real [getattr](file:///e:/Dev/Filmu/FilmuCore/filmu_py/core/byte_streaming.py#339-343) — use existing [getattr_for_path_id](file:///e:/Dev/Filmu/FilmuCore/filmu_py/core/byte_streaming.py#339-343) wrapper.
5. Real [release](file:///e:/Dev/Filmu/FilmuCore/filmu_py/core/byte_streaming.py#648-652) — close handle, clean up stream, update governance counters.

### Phase 3 — Integrate the shared chunk engine + caching (P2)

- wire the shared engine into direct-file/HTTP byte reads

## Summary Verdict

| Question                                      | Answer                                                                                                                                                                                                                              |
| --------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Is FilmuCore running the full pipeline today? | **Partially.** Real scrape → parse → rank → debrid → finalize stages exist with built-in scraper and provider clients. Items can progress through the full lifecycle. Missing: Plex/Emby VFS playback validation and content-source webhooks. |
| Is FilmuCore designed to do so?               | **Yes.** The ARQ pipeline, RTN ranking, debrid clients, domain model (15+ ORM tables), Rust VFS sidecar, and shared serving substrate are all production-shaped.                                                                    |
| Can FilmuCore exceed Riven-TS?                | **Yes, and it already leads in several areas.** REST breadth, SSE surface, serving observability, debrid provider breadth (3 vs 1), plugin capability model, and config flexibility. The remaining gap is productionizing the mounted VFS path and completing Plex/Emby validation. |
| What is the most critical missing piece?      | **Plex/Emby validation plus mount-side chunk-engine adoption.** The pipeline itself is real, the Rust sidecar now mounts and reads real files under WSL/Linux, and the next step is turning that into a fully validated playback path. |
