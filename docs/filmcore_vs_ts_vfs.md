## TL;DR

The **current local TypeScript backend** VFS is a **complete, running FUSE mount** with chunk caching, read-type detection, and BullMQ-backed link resolution. FilmuCore's VFS is no longer only a substrate: it now has a real Unix-only Rust sidecar mount loop with chunk-engine-backed mounted reads that has been exercised in WSL/Linux through automated lifecycle validation, manual mounted-read smoke, and Plex/Emby playback validation. The remaining gap is no longer first mount bring-up or first playback success; it is productionizing the mounted path with stronger path semantics, queue-backed resolver dedup, richer observability, and longer-running soak/backpressure hardening.

April 2026 local re-audit note:

- the local `Triven_backend - ts` comparison checkout is now clean and aligned directly to upstream `rivenmedia/riven-ts` `main` at `e64604a`
- the current baseline already includes `fuse-error.ts`, `with-vfs-scope.ts`, sandboxed-job files, newer main-runner actors, and the broader CI/observability scaffolding
- current `main` still keeps `@zkochan/fuse-native`; the Rust daemon work has not replaced the in-process VFS on `main`

## March 2026 upstream `feat/rust-vfs` note

The upstream [`riven-ts`](https://github.com/rivenmedia/riven-ts/tree/feat/rust-vfs) `feat/rust-vfs` branch should **not** currently be treated as a feature-complete replacement for the local TS backend used as the baseline in this document.

It adds a promising Rust-daemon + gRPC contract split, but it currently removes the richer local TS `apps/riven/lib/vfs/**` behavior before reintroducing equivalent path modeling, read heuristics, chunk reuse, and config/documentation alignment.

See [`RIVEN_TS_RUST_VFS_BRANCH_AUDIT.md`](RIVEN_TS_RUST_VFS_BRANCH_AUDIT.md) for the verified branch-vs-local audit.

## March-April 2026 correction

The matrix below predates the implementation of the shared Python chunk engine in [`../filmu_py/core/chunk_engine.py`](../filmu_py/core/chunk_engine.py).

That means the remaining delta is no longer “build any chunk engine at all.” It is:

- preserve one shared geometry/classification model across Python and Rust surfaces
- keep converging the Python HTTP byte path and the Rust mounted read path semantically
- harden on-disk caching, prefetch, abort behavior, and observability above the now-working mounted/runtime core

## Structural Comparison

### How FilmuCore approaches things better

**1. Session/handle/path accounting is first-class**  
TS tracks fds with flat `Map` globals per operation file. FilmuCore has named [ServingSession](file:///E:/Dev/Filmu/FilmuCore/filmu_py/core/byte_streaming.py#58-69), [ServingHandle](file:///E:/Dev/Filmu/FilmuCore/filmu_py/core/byte_streaming.py#71-85), [ServingPath](file:///E:/Dev/Filmu/FilmuCore/filmu_py/core/byte_streaming.py#87-99) dataclasses with owners, timestamps, bytes_served, and retention policies. This means audit, stale cleanup, and VFS/HTTP unification happen at the substrate layer, not scattered across FUSE op handlers.

**2. Owner-aware retention**  
`future-vfs` handles get 1-hour retention; HTTP handles get 15 minutes. TS has no equivalent concept — mount handles live until FUSE [release](file:///E:/Dev/Triven_riven-fork/Triven_backend/src/program/services/filesystem/vfs/rivenvfs.py#2034-2066) is called.

**3. HLS is already designed into the shared engine**  
The design principle in [FILMUVFS_BYTE_SERVING_PLATFORM_MATRIX.md](file:///E:/Dev/Filmu/FilmuCore/docs/TODOS/FILMUVFS_BYTE_SERVING_PLATFORM_MATRIX.md) explicitly says: "Build one shared byte-serving engine, then expose it through multiple surfaces." HLS, local file, and remote proxy all go through the same session/handle tracking. TS has no HLS at all.

**4. Attachment abstraction handles heterogeneous metadata**  
[PlaybackAttachment](file:///E:/Dev/Filmu/FilmuCore/filmu_py/api/playback_resolution.py) + [extract_sources()](file:///E:/Dev/Filmu/FilmuCore/filmu_py/api/playback_resolution.py) can resolve from any flexible JSON shape — no schema contract required yet. This is a compatibility bridge that TS doesn't need because its domain model is stricter.

**5. Status endpoint**  
`GET /stream/status` exposes live sessions, handles, paths, and governance counters. TS has no monitoring surface for VFS state.

### How TS approaches things better

**1. The FUSE mount is actually running**  
TS has [open](file:///E:/Dev/Triven_riven-fork/Triven_backend/src/program/services/filesystem/vfs/rivenvfs.py#1705-1807), [read](file:///E:/Dev/Triven_riven-fork/Triven_backend/src/program/services/filesystem/vfs/rivenvfs.py#1808-2033), [readdir](file:///E:/Dev/Triven_riven-fork/Triven_backend/src/program/services/filesystem/vfs/rivenvfs.py#1656-1704), [getattr](file:///E:/Dev/Triven_riven-fork/Triven_backend%20-%20ts/apps/riven/lib/vfs/operations/getattr.ts#134-329), [release](file:///E:/Dev/Triven_riven-fork/Triven_backend/src/program/services/filesystem/vfs/rivenvfs.py#2034-2066) wired to a real FUSE mount. FilmuCore now also has a real Unix-only Rust mount loop with mounted read execution, but it is still earlier-stage operationally: it now passes WSL/Linux lifecycle validation, manual mounted-read smoke, and Plex/Emby playback validation, while longer-running runtime hardening still remains.

**2. Read-type classification**  
TS's [detectReadType](file:///E:/Dev/Triven_riven-fork/Triven_backend%20-%20ts/apps/riven/lib/vfs/utilities/detect-read-type.ts#15-66) (header-scan, footer-scan, footer-read, body-read, cache-hit) is a mature heuristic for media streaming patterns. FilmuCore now has the equivalent classification model in [`../filmu_py/core/chunk_engine.py`](../filmu_py/core/chunk_engine.py), plus a mounted Rust chunk-engine path. The remaining gap is cross-surface semantic alignment and parity hardening rather than first read-pattern implementation.

**3. Discrete chunk cache**  
TS caches by `"${fileName}-${start}-${end}"` — so header probes, footer reads, and body chunks are independently cacheable and reusable across multiple file descriptors. FilmuCore now also has a byte-weighted per-range in-memory cache via [`ChunkCache`](../filmu_py/core/chunk_engine.py) for the HTTP path, plus memory and hybrid cache engines in the Rust mounted path. The remaining gap is validation, observability, and policy hardening rather than first cache implementation.

**4. Link resolver is a proper service**  
TS uses BullMQ queues per provider with `fileNameIsFetchingLinkMap` dedup. FilmuCore now resolves URLs/attachments through [playback_resolution.py](file:///E:/Dev/Filmu/FilmuCore/filmu_py/api/playback_resolution.py), [playback.py](file:///E:/Dev/Filmu/FilmuCore/filmu_py/services/playback.py), and built-in Real-Debrid / AllDebrid / Debrid-Link clients in [debrid.py](file:///E:/Dev/Filmu/FilmuCore/filmu_py/services/debrid.py). It also now has a first provider-side attachment projection model so download-id refreshes can persist matched provider file identity instead of depending only on filename/filesize heuristics. The remaining gap is queue-backed dedup, broader provider-download-id refresh workflows, and a richer VFS-facing resolver workflow that turns those projections into durable read models.

**5. Path type schema**  
`PathInfo` Zod schema gives TS structured extraction of `tmdbId`, `tvdbId`, `season`, `episode` from the VFS path. FilmuCore's path categories (`local-file`, `remote-proxy`, `generated-hls`) are serving-surface categories, not media-semantic path types. Browsing `/shows/tt1234567/Season 2/` requires the latter.

**6. Abort-safe reads**  
TS [read](file:///E:/Dev/Triven_riven-fork/Triven_backend/src/program/services/filesystem/vfs/rivenvfs.py#1808-2033) catches `RequestAbortedError` and returns `0` cleanly. FilmuCore now has a mounted read path, but the exact TS-style zero-byte abort semantics and broader cancellation policy still need deliberate hardening.

**7. Hidden path filtering**  
TS silently eats macOS probes (`.DS_Store`, `.Spotlight-V100`, `._*`). FilmuCore now also implements hidden/ignored path filtering in the Rust sidecar; the remaining gap is broader path-semantics depth rather than first probe suppression.

The basic [`getattr`](file:///E:/Dev/Triven_riven-fork/Triven_backend%20-%20ts/apps/riven/lib/vfs/operations/getattr.ts#134-329), [`readdir`](file:///E:/Dev/Triven_riven-fork/Triven_backend/src/program/services/filesystem/vfs/rivenvfs.py#1656-1704), [`open`](file:///E:/Dev/Triven_riven-fork/Triven_backend/src/program/services/filesystem/vfs/rivenvfs.py#1705-1807), [`read`](file:///E:/Dev/Triven_riven-fork/Triven_backend/src/program/services/filesystem/vfs/rivenvfs.py#1808-2033), and [`release`](file:///E:/Dev/Triven_riven-fork/Triven_backend/src/program/services/filesystem/vfs/rivenvfs.py#2034-2066) equivalents now exist in the Rust sidecar.
